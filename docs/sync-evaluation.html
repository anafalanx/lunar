Lunar — Deep Evaluation of the Sync & Adjustment Mechanism

Security · Guaranteed correctness · Availability — proposal, no code changes.

TL;DR

Lunar's clock refuses to display the Windows system time; it runs its own disciplined UTC from four parallel Internet sources, gated by a cryptographic anchor (NTS) and a concurrence check against three stratum-1 core servers. The design is sound and fail-safe, but has seven security weaknesses, seven correctness gaps, and six availability liabilities. The highest-leverage fixes — tiered trust with graceful degradation, anchor monotonicity, per-cycle rate-change clamp, and an adversarial-input test suite — are ~150–400 LOC each and should land before any availability work.

Primer 0 — orientation

Lunar is a desktop analog clock that refuses to trust the Windows system clock. The Windows clock can be wrong for many reasons: user edit, malware, CMOS battery, drift, broken w32time, VM sleep/wake. Lunar reads the true time directly from Internet time servers, keeps its own monotonic running clock locally, and renders a red INOP state whenever it cannot verify the time. The pipeline analysed below is the safety-critical core: what lets the clock claim "trusted time" and what lets it refuse to lie.

1. Current design (as built)

Sources, per cycle (fired in parallel)

  • Slots 0-3 — four core SNTPv4 sources over UDP/123, randomly drawn from the curated metrology/research pool; each slot gets one same-cycle replacement attempt if its first server fails
  • Slots 4-5 — two NTS-authenticated sources, randomly drawn from the NTS provider metadata pool with a preference for distinct operator families; each slot uses TLS 1.3 + ALPN ntske/1, authenticates through a local enrolled SPKI pin or Windows CA enrollment/renewal, and then performs authenticated SNTP

Primer 1 — NTP · SNTP · NTS · SPKI · KE

NTP (RFC 5905)
Network Time Protocol. Client timestamps its request (T1), server stamps receipt (T2) and reply (T3), client stamps arrival (T4). Round-trip and offset follow from these four numbers.
SNTP (RFC 4330)
Simple NTP: stateless one-shot subset. No long-running association, no client-side statistical filtering. Adequate for polling, which is what Lunar does.
NTS (RFC 8915)
Network Time Security. A one-time TLS-1.3 key establishment (KE) gives the client cookies/keys; subsequent UDP time packets are authenticated with AEAD so an on-path attacker cannot forge or tamper with them.
SPKI pinning
Subject Public Key Info. Lunar stores the SHA-256 hash of an endpoint's certificate public key after Windows CA enrollment/renewal. Ordinary operation uses that local continuity pin; renewal re-enters the Windows/Web PKI.
UDP port 123
Standard NTP port. Firewalls and captive portals frequently block or rewrite it.

Trust gate

Both NTS slots must succeed, both must be authenticated by enrolled pins, they must come from different operator families, and they must mutually agree within ±200 ms. At least 3 of 4 core sources must project to within ±200 ms of the NTS midpoint on a common QPC moment. Otherwise the verdict is TRUST_INOP, Clock_NowUtcMs() refuses to return a time, and the UI renders red INOP.

Anchor

On an OK cycle: the midpoint of the two agreeing operator-diverse NTS samples, projected to the selected QPC moment.

Rate discipline

Per-cycle observed ppm over the inter-sync interval, clamped to ±500 ppm (single-cycle target up to ±2000 ppm before clamp), EMA α = 0.25, persisted to %APPDATA%\Lunar\discipline.dat at shutdown, reloaded as bootstrap, rejected at > 30 days against disciplined UTC.

Residual handling

|residual| ≤ 2 s → slew linearly over 60 s. > 2 s → instant snap (anchor rebase).

Cadence

60 s on TRUST_OK; 5 s on TRUST_INOP.

Safeguards

SNTP header validation (LI / VN / mode / stratum 1–15); cross-check that a claimed-OK cycle matches the running projection within 200 ms — else "local-oscillator fault" → INOP.

Primer 2 — QPC · UTC · PLL · EMA · ppm · anchor · slew · snap

QPC
QueryPerformanceCounter: Windows' monotonic hardware tick counter. Never moves backwards, immune to wall-clock edits, fixed frequency (typically 10 MHz). Lunar uses QPC for all local timing; it uses server replies only for UTC.
UTC
Coordinated Universal Time. Represented in Lunar as milliseconds since the Unix epoch.
PLL
Phase-Locked Loop. Feedback that steers a local oscillator to a reference. Phase = instantaneous time; frequency = rate. Lunar's PLL fires once per cycle and adjusts local rate (ppm) so future QPC ticks project onto correct UTC.
EMA
Exponential Moving Average. new = old + α·(sample − old). Lunar uses α = 0.25. Lower α → more damping, slower response; higher α → noisier, faster.
ppm
Parts per million. +57 ppm = 57 µs fast per second = 4.9 s/day. Typical quartz drift is ±50 ppm.
anchor
A pair (qpc, utc) that pins "this QPC moment corresponded to this UTC". Projection: utc + (qpc_now − qpc_anchor) · (1 + ppm/1e6).
slew vs snap
Slew = apply a small correction gradually over a window so the second hand stays smooth. Snap = rebase the anchor instantly, which makes the clock jump but is the only sane option for large errors.
INOP
Aviation-inspired: when the instrument cannot guarantee correct data, it says so instead of guessing.

2. Security analysis

Threat model (implicit today)

Protected against: passive on-path observer, single-ISP hijack of UDP/123, DNS poisoning of one source, rate-limiting or outage of one source. Relies on NTS as the cryptographic trust anchor.

Primer 3 — trust anchor · threat model · attacker types

Trust anchor
The element whose integrity, if preserved, guarantees the whole verdict. For Lunar: the NTS reply. Even if all three plaintext SNTP sources were hijacked, disagreement with NTS would trip INOP.
On-path attacker
Sits between client and server; can read, drop, modify, inject (hostile ISP, malicious Wi-Fi, state filter). Off-path attacker can only inject without seeing — harder; needs to guess response fields. NTS defeats both for the time payload; plain SNTP defeats neither.
Threat model
Explicit enumeration of adversaries, capabilities, and attack goals the design intends to defeat. "Implicit today" means Lunar has no written one; it is inferred from the code.

Weaknesses

S1NTS is a single point of cryptographic trust — silently

If the NTS slot fails for any reason (KE timeout, pin mismatch, provider outage), the gate goes INOP. Correct fail-safe, but an adversary who selectively blocks TLS to all NTS providers while letting UDP/123 through keeps the clock indefinitely INOP — a denial-of-availability attack via the security layer. There is no "NTS-disabled, degraded-trust fallback" — INOP is permanent until NTS recovers.

S2Core sources use hardcoded hostnames, no DNSSEC

A resolver-level attacker can return any IP. They can't forge agreement with NTS (defeated at the concurrence gate) but they can deny a core source (drop/RST) or flood the socket budget with garbage. Combined with mild NTS degradation, this widens the DoS surface.

Primer 4 — DNS · DNSSEC · TCP RST · rate-limiting

DNS
Domain Name System. Resolves hostnames to IPs; normally unauthenticated. A hostile resolver can point any name anywhere.
DNSSEC
Cryptographic signing of DNS. Windows' default resolver does not validate end-to-end; most apps (including Lunar) inherit that gap.
TCP RST
A reset packet that forcibly closes a connection. Selective RST injection can break TLS handshakes one at a time.
Rate-limiting
Public time servers throttle or ban clients that poll too often — relevant to finding A3.

S3NTS provider pool rotation is random, not load-shed-aware

If one pinned provider serves stale or bad data (compromised or misconfigured), it gets picked ~1/N of cycles. There is no penalty or blacklist for a pin that recently produced an outlier that tripped the local-oscillator fault check. One bad provider keeps cycling back in.

S4The 200 ms local-oscillator-fault check is symmetric with the concurrence threshold

Elegant — but it means an attacker who can induce a ~150 ms NTS offset (within gate) that is consistent with two hijacked core sources will succeed. The gate is "≥ 2 core agree with NTS within 200 ms", not "…within a tighter bound". Because NTS is authenticated, the practical threat is provider compromise; but the gate architecture does not distinguish "all four sources agree to 20 ms" (high confidence) from "scraped agreement at 199 ms" (suspicious).

S5Persisted rate file is unauthenticated

%APPDATA%\Lunar\discipline.dat is plain ASCII "<ppm> <lastSyncUtcMs>\n". A local attacker (or ransomware) that can write AppData can inject ±500 ppm, applied as bootstrap until the first sync re-verifies it. Window: up to ~6 s of wrong rate on first-anchor acquisition, which then snaps. Low impact, but the "safety clock" invariant argues for at least a MAC or checksum.

Primer 5 — MAC · HMAC · DPAPI

MAC
Message Authentication Code. A short tag computed from message + secret key; detects tampering by anyone without the key.
HMAC-SHA256
A well-analysed MAC construction on top of SHA-256.
DPAPI
Windows Data Protection API (CryptProtectData). Encrypts a blob under a key derived from the user's logon credentials. Simpler than managing our own keys; right-sized for "protect from other local users and offline attackers".

S6No NTS KE replay / freshness discipline beyond TLS 1.3

We rely on TLS 1.3 client-hello entropy + NTS cookies for replay protection. Correct by spec, but there is no independent freshness bound: if an attacker can force TLS session resumption with a stale ticket (on a provider that mis-implements ticket rotation), the returned timestamp could predate the attack. Low likelihood — but verifiable by having clock.c reject any ntpUtcMs that decreases by > X s from the previous OK cycle.

Primer 6 — TLS 1.3 replay · session resumption · 0-RTT

Replay
Capture a valid encrypted message and re-send later. TLS 1.3 defends handshakes with fresh randoms and records with AEAD nonces.
Session resumption / 0-RTT
For fast reconnects, TLS 1.3 lets a client attach application data to the first flight using a pre-shared key from a prior session. 0-RTT data is not spec-guaranteed non-replayable — the application must cope. NTS handles this with cookies; server-side mis-implementations have been seen.

S7No bound on single-cycle rate target

corr is clamped to ±2000 ppm before the EMA, so one malicious-but-gate-passing cycle can pull the EMA by up to (2000 − current)·0.25. Over 2–3 carefully-shaped cycles, an adversary who can defeat the gate once could drive the persisted ppm to ±500 (the hard clamp). The "pulse a cycle, wait, pulse again" variant. Defense: require N consecutive OK cycles before a rate update persists to disk, plus a per-cycle change clamp (|Δppm| ≤ 20 per cycle).

3. Correctness analysis

Primer 7 — monotonicity · projection · RTT asymmetry

Monotonicity
A clock is monotonic if it never moves backwards. Displayed analog clocks that jump backwards are disconcerting and, in a safety context, dangerous (events get reordered).
Projection
Given two samples at different QPC moments, "project" means advance or retard one by elapsed QPC × current rate so the two become comparable.
RTT asymmetry
NTP assumes forward and return legs take equal time. On transoceanic or multi-ISP paths this is often false; the error is a constant bias in ntpUtcMs equal to half the asymmetry.

C1First-anchor trust escape

On the very first OK cycle we accept (ntpUtcMs, qpcAtT4) verbatim — no prior anchor exists to cross-check. Combined with a stale persisted rate (up to +57 ppm seen in the live log), the first ~10 s of displayed time can be off by "rate × interval since anchor" before the cycle-2 slew corrects it. For a safety-critical clock this is a visible defect: the moment the user opens the app is the moment the clock is least trustworthy.

C2RTT asymmetry assumption

ntpUtcMs = t3 + netRtt/2. Asymmetric routing makes the error up to netRtt/2 — for NICT (rtt ≈ 300 ms) that is ±150 ms worst case. The 200 ms gate barely covers this; happy-path spread is 14 ms, but on a bad day NICT alone could push us over the gate, forcing INOP.

C3EMA on ppm, but rate samples are not IID

A short interval plus a large residual produces a huge corr. The elapMs > 30000 guard prevents the trivial blow-up but does nothing about the much more common "85 s cycle, 9 ms residual" → corr ≈ 106 ppm, pulled by α = 0.25 → +26 ppm jump. The live log shows ±8–34 ppm jumps between consecutive cycles — the PLL chasing network jitter, not drift. Proper approach: a PI controller or small Kalman filter with an explicit measurement-noise variance. Chronyd and NTPsec have 20 years of tuning we could borrow.

Primer 8 — IID · jitter · PI controller · Kalman filter

IID
Independent and Identically Distributed. A simple EMA implicitly assumes this; NTP samples are not IID (shared route state, server load, …), which makes a naive EMA noisy.
Jitter
Short-term variability in RTT or offset. Distinct from drift, which is long-term monotonic deviation of the oscillator.
PI controller
Proportional-Integral. u = Kp·error + Ki·∫error dt. Proportional rejects immediate error; integral removes steady-state bias. Standard for clock-discipline loops; tune Kp/Ki for bandwidth and damping.
Kalman filter
Optimal (under Gaussian noise) state estimator tracking phase, frequency, and optionally drift, with each measurement's covariance. Strictly more powerful than PI but more parameters to get right.
chronyd · NTPsec
Two well-maintained open-source NTP clients; their disciplining loops are documented and battle-tested.

C4Slew is linear; clocks run smooth but non-monotonic

If a new slew overrides an in-progress slew, the displayed time can briefly go backward. Slew-over-slew should either (a) complete the old slew into the anchor first, then start the new one, or (b) superimpose slews. Today neither happens — the old slew is discarded mid-flight, leaving a small residual that the next sync rediscovers.

C5No sanity bound on anchor jump

An NTS provider returning 2020 would be accepted without cross-check against the previous anchor. The 200 ms concurrence gate requires core agreement, so in practice this is caught — but if 2-of-3 core sources also return 2020 (hypothetical coordinated attack or cached resolver result) we snap to 2020 silently. A monotonic bound "new anchor cannot predate the last OK anchor by more than (elapsed_qpc + 60 s slack)" would catch this class deterministically.

C6Rate clamp is rectangular

Real crystal oscillators drift ~±50 ppm; ±500 ppm is 10× observed. A target beyond ±100 ppm is almost certainly an error (bad sync pair, sleep-wake with stale QPC, malicious sample). The clamp should be narrower by default and widen only on confirmed sustained drift.

C7No test coverage of the adjustment math under adversarial inputs

Tests verify the concurrence gate and projection arithmetic, but nothing feeds Clock_OnSyncedNtpUtc a time series with injected jitter / bias / jumps and asserts stability. For a safety clock this is the single biggest gap.

4. Availability analysis

Primer 9 — availability · DoS · back-off · dual-stack

Availability
Fraction of time the service performs its intended function. For Lunar: fraction of wall-clock time during which the UI can render a trusted time (green) rather than INOP (red).
DoS
Denial of Service. Reduces availability without necessarily compromising integrity. Lunar's fail-safe stance turns many integrity threats into DoS problems — correct, but makes DoS the dominant failure mode.
Back-off
Lengthening retry interval when failures persist, to avoid hammering a degraded service. Canonical pattern: exponential with jitter.
Dual-stack
Supporting both IPv4 and IPv6 transparently, so hosts reachable only over one still work.

A1Hard dependency on NTS availability

See S1. Real-world NTS providers (Cloudflare, Netnod, SIDN, PTB, NetTime) have had multi-hour outages; a coincident outage of two providers within our small pool puts us permanently INOP despite three healthy NIST/PTB/NICT sources.

A2Aggregator wait is WaitForMultipleObjects(..., TRUE, 20000)

If any worker is stuck (e.g. a socket call not honoring its timeout due to a Winsock driver bug), the entire cycle blocks for 20 s before the others are harvested. Core workers that completed in ~300 ms sit idle. Fix: wait with WAIT_ALL=FALSE and poll, or wait per-source with individual deadlines.

A35 s cadence during INOP can DoS the providers

If NIST and NICT are both degraded, we hammer them every 5 s forever. Friendly to us (fast recovery) but hostile to public infrastructure; may earn rate-limiting and prolong INOP. Recommendation: exponential back-off on INOP (5 → 10 → 20 → 60 s cap), reset to 5 s on first OK.

A4No caching of "last known good anchor" for display-only fallback

Architecturally deliberate (we promise INOP when untrusted), but a user-facing concession would be: during INOP, render the last-known-good time in amber with "LAST KNOWN: 12 s ago" — still safer than the system clock, still visually distinct from trusted green, and keeps the clock usable during brief network hiccups.

A5Cold-start latency

First OK cycle observed at T+6.3 s. That is the user's entire first-impression window rendering INOP. Pre-flighting a fast SNTP-only query against a cached IP (previous cycle's resolved address) while the NTS handshake catches up would shorten this considerably.

A6No IPv6

hints.ai_family = AF_INET. On IPv6-only networks (increasingly common in mobile and enterprise) the core sources fail hard.

5. Proposal (prioritized)

Tier 1 — security & correctness (ship first)

  1. Tiered trust with graceful degradation (addresses S1, A1).
    • TRUST_OK — NTS + ≥ 2 core within 200 ms (today).
    • TRUST_DEGRADED — NTS unavailable, but ≥ 3 core sources concur within 100 ms (tighter) and last NTS-OK is < 2 h old. Clock runs disciplined; UI shows amber "UNAUTHENTICATED"; rate is read-only (no EMA update from degraded cycles).
    • TRUST_INOP — nothing else.
    Availability rises; security stance is explicit (degradation is user-visible, not silent); PLL is protected from being poisoned by unauthenticated samples.
  2. Anchor monotonicity guard (C5, S6). Reject any new anchor where ntpUtcMs < previousAnchorUtcMs + elapsedQpcMs − slack (slack ≈ 60 s). Hard reject: INOP, log clock: anchor rejected — retrograde time (Δ=…).
  3. Per-cycle rate-change clamp (S7, C3). Limit |newRate − oldRate| ≤ 20 ppm per cycle regardless of EMA math. Sustained drift still converges in ~ 5 cycles; a single adversarial pulse cannot swing the rate by > 20 ppm.
  4. Disk integrity on discipline.dat (S5). Wrap with DPAPI (CryptProtectData). Reject tampered files, fall back to 0-ppm bootstrap, log.
  5. Slew superposition (C4). On new residual during active slew, fold the remaining residual of the old slew into the anchor first, then start the new slew. Guarantees the displayed clock is monotonic within the slew window.
  6. Adversarial-input test suite (C7). Table-driven tests that feed Clock_OnSyncedNtpUtc crafted sequences: steady drift, step change, sleep-wake (QPC gap), alternating-bias attack, slow bias-drift attack. Assert: monotonicity, convergence, bounded rate excursion, no snap-storm.

Tier 2 — availability

  1. Exponential INOP back-off (A3): 5 / 10 / 20 / 40 / 60 s, reset on OK.
  2. Parallel-harvest aggregator (A2): decouple per-source deadlines so core replies are processed the instant they arrive; NTS runs on its own 20 s budget.
  3. IPv6 dual-stack (A6): AF_UNSPEC + try-all.
  4. Last-known-good amber mode (A4): render recent-but-stale time distinctly, not INOP, for outages ≤ 2 min. Guarded by the TRUST_DEGRADED tier above.
  5. Warm-start pre-flight SNTP (A5): one unauthenticated query against the cached IP of the fastest previous source, fired in parallel with Init. Displayed as "SYNCING" (untrusted) until the first gated OK lands.

Tier 3 — defense-in-depth

  1. Confidence-weighted concurrence. Gate still passes at 200 ms, but log confidence = 1 − (spread/200); display HIGH / MED / LOW in About. Suspicious-but-passing cycles (< 50 % for N consecutive cycles) raise a user-visible warning without forcing INOP.
  2. NTS pool health tracking. Per-provider EWMA of RTT, handshake success rate, outlier rate. Provider weight ∝ health; a recently-faulting provider gets 0 weight for N cycles.
  3. Replace ppm-EMA with PI controller (or small Kalman) (C3). Well-understood literature; chronyd is the reference. Lower steady-state wander, faster convergence, rejects measurement noise better.
  4. Kiss-o'-Death handling. We reject stratum 0 but do not parse the reference-ID kiss code. Respect RATE / DENY; apply per-provider cooldown.
  5. Add a 5th slot: Roughtime (future). Independent protocol (Cloudflare / Google). Converts NTS from single cryptographic anchor to 2-of-2.

Primer 10 — Kiss-o'-Death · Roughtime · EWMA

Kiss-o'-Death (KoD)
An NTP server reply with stratum 0 and a 4-letter reference-ID code (RATE, DENY, …) asking the client to stop or slow down. Respecting these keeps us in good standing with public servers.
Roughtime
Independently-designed secure time protocol using Ed25519 signatures with explicit "I will misbehave — here is the proof" auditability. Complements NTS: different crypto stack, different operator ecosystem.
EWMA
Exponentially Weighted Moving Average. Same as EMA; the term is common in quality-control and health-monitoring contexts. Used here for per-provider health scores.

6. Recommendation

Recommended first sprint

Start with Tier 1 items 1–3 + 6 (tiered trust, anchor monotonicity, rate-change clamp, adversarial test suite). These address the highest-severity findings (S1, S7, C3, C5, C7), each is ~150–400 LOC, and the test suite is the investment that makes every subsequent change safe.

Tier 2 availability work should be gated behind Tier 1: "improving availability" without tighter safety gates would amplify any latent correctness bug.

Tier 3 is long-term hygiene — one item per iteration once Tier 1 and Tier 2 have landed.
